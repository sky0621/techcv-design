# 認証方式の採用理由

## 採用方針

- MVPでは Cookie ベースのセッション認証を採用する
- API仕様上は Bearer Token も将来拡張の余地として残す
- ただし初期実装の主軸は SessionAuth とする

## なぜセッション認証を選ぶか

### 1. 今回の利用形態と相性がよい

- このプロダクトは個人のWebエンジニア本人が利用するWebアプリである
- MVP時点では外部クライアント連携や公開API提供を前提にしていない
- そのため、ブラウザ中心の利用に適したセッション認証が自然である

### 2. セキュリティ実装を単純にしやすい

- `HttpOnly` Cookie を使うことで、フロントエンドから認証情報を直接触らせずに済む
- `Secure` と `SameSite` を組み合わせることで、基本的な安全性を確保しやすい
- ローカルストレージなどへのトークン保存を避けられるため、設計上の事故を減らしやすい

### 3. MVPで必要な複雑さに対して過不足がない

- JWT/Bearer Token を主軸にすると、保存場所、期限切れ、再発行、更新トークンなどの論点が増える
- MVPではそこまでの複雑さを先に抱えるメリットが小さい
- まずはログイン、ログアウト、セッション維持、`/auth/me` 取得が安定していれば十分である

### 4. 個人情報を扱う用途と相性がよい

- 経歴書管理システムでは、氏名、連絡先、職務経歴などの個人情報を扱う
- MVPでは利便性よりも、まず安全に非公開情報を扱えることを優先したい
- セッション認証はそのための実装コストと安全性のバランスが良い

## JWT/Bearer Token を主軸にしない理由

- MVP時点ではモバイルアプリ対応を前提としていない
- サードパーティ連携や外部利用者向けAPIを公開する予定がない
- SPA + API 構成でも使えるが、トークン管理の設計負荷が増える
- 将来必要になってから追加しても遅くない

## 想定する実装方針

- ログイン成功時にサーバー側でセッションを発行する
- 認証情報は `HttpOnly` `Secure` `SameSite=Lax` の Cookie で保持する
- `POST /auth/logout` でセッションを破棄する
- `GET /auth/me` で現在のログイン状態を確認する
- 状態変更系エンドポイントには CSRF 対策を入れる
- パスワードは平文保存せず、適切にハッシュ化して保存する

## OpenAPI上の扱い

- `openapi.yaml` では `BearerAuth` と `SessionAuth` の両方を定義している
- これは将来拡張の余地を残すためである
- 実装上の優先順位は `SessionAuth` を先とする
- 将来、モバイルアプリや外部APIが必要になった場合に Bearer Token を追加検討する

## Bearer Token を本格採用する判断条件

- ネイティブアプリを提供する
- 外部クライアント向けAPIを提供する
- 別ドメインのフロントエンドを強く前提にする
- OAuth/OIDC 連携を早い段階で必要とする

## 結論

- 現時点のMVPにはセッション認証が最も適している
- 将来拡張を見据えて OpenAPI 上は両対応の余地を残す
- 実装は SessionAuth 先行で進める
